# TRADE COURT — ロードマップ

これから何をするか。日ごとの記録は [worklog.md](worklog.md)、
今どう作られているかは [design.md](design.md)。

## 進め方の方針

- 「動く小さな塊」を積み上げる。各段階で `rails console` や画面で動作を確認する
- 変更コストの高いもの（データ構造）から先に、安いもの（デザイン・ログイン）を後に
- 学習プロジェクトなので、コードには日本語コメントを厚めに入れる
- 1機能 = 1ブランチ = 1PR。CI が green になってから main にマージする
- 公開して他人の投稿が入るまでは、データ構造をいくらでも作り直せる

## 決定事項

| # | 決めたこと | 理由 |
|---|---|---|
| 1 | Rails 8.1.3 / Ruby 3.4.10 / SQLite / Docker | 学習しやすさ。ホストを汚さない |
| 2 | ポートはホスト側 3001 | 3000 は別プロジェクトが使用中 |
| 3 | 欲しい／譲れるを1投稿に統合 | トレードは求と出がセット。分けると結びつかない |
| 4 | 弾＝掲示板、運営が事前に用意する | 過疎に見えるのを防ぐ |
| 5 | チームもモデルにして掲示板の入口にする | 弾と対等に画像・並び順を持たせるため |
| 6 | 欲しい／出せるとも複数選択・検索式 | 「出せる人」も探せるようになる。UIも対称になる |
| 7 | 募集は弾に所属させない | 欲しいカードが複数弾にまたがりうる。掲示板は欲しいカードから導く |
| 8 | 同一選手・同一弾の複数カードは name で区別 | 得点記録／アシスト記録 が別カードとして存在する |
| 9 | レア度は数字1〜5、画面は★、当面4〜5だけ扱う | 大小比較・絞り込みができる。器は広く中身は絞る |
| 10 | トレード成立ボタン → 非表示、成立済みはタブで閲覧 | 終わった募集を消しつつ、レート表として記録を残す |
| 11 | アカウントID は公開しない | 晒しに悪用される。成立済みでは特に不要 |
| 12 | ログインは投稿必須・閲覧自由。後付け | 気軽さと荒らし対策の両立 |
| 13 | 弾は YAML の末尾に追記、画面では新しい順 | 運用が自然な順序のまま、表示だけ逆順にする |
| 14 | 画像は公式素材のため git 管理外 | 著作権に配慮。ローカルに置けば表示される |
| 15 | robots.txt 全面ブロック（当面） | 公開判断まで検索インデックスを避ける |
| 16 | 連投対策は `rate_limit` | 構造化フォームだけでは荒らしを防げない |

## 未決事項

Phase 7（公開）までには判断が要るもの。

- **公開範囲**: 自分の練習用か、実際にネット公開して他の人にも使ってもらうか
  → これが下記すべての前提になる
- **DB**: SQLite のままか PostgreSQL に移行するか（公開して投稿を残すなら要判断）
- **画像の著作権**: 弾のバナー・チームロゴは B.LEAGUE 側の素材。公開・収益化するなら
  ガイドライン確認または問い合わせが要る。ロゴは商標でもあり、特に慎重に
- **収益化**: 広告を入れるかどうか。他社IPの上に成り立つサービスなので、
  営利化はリスクが上がる（詳細は会話の記録を参照）
- **不適切な投稿への対応**: 通報・削除の仕組みをどこまで作るか
- **トレードの安全性**: 成立後の評価・信頼の仕組みを入れるか

## 注意メモ

- `config/master.key` は git 管理外。**失うと credentials.yml.enc を復号できない**。
  本物の秘密を入れる前・公開する前に安全な場所へ控えること
- 画像（`app/assets/images/sets/`, `teams/`）は git 管理外。
  デプロイ時は別途サーバーに置く必要がある
