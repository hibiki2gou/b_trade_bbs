# B-Trade BBS — ロードマップ

## 進め方の方針

- 「動く小さな塊」を積み上げる。各段階で `rails console` や画面で動作を確認する
- 変更コストの高いもの（データ構造）から先に、安いもの（デザイン・ログイン）を後に
- 学習プロジェクトなので、コードには日本語コメントを厚めに入れる
- 公開して他人の投稿が入るまでは、データ構造をいくらでも作り直せる

---

## フェーズ

### Phase 0 — 環境構築 ✅ 完了

- Rails 8.1.3 / Ruby 3.4.10 / SQLite / Docker
- `docker compose up` → http://localhost:3001
- 初回コミット・GitHub 公開済み

### Phase 1 — モデルとマスタデータ ✅ 完了

- Board / Topic / Player / Card / Post の5モデル
- マイグレーション（制約・デフォルト・重複禁止）
- 関連・バリデーション・成立ボタンを console で検証
- `db/seeds/card_sets.yml` から検証つき・冪等に投入
- robots.txt を全面ブロック

### Phase 2 — 画面（次にやる）

- [ ] トップ（カテゴリ一覧）
- [ ] 弾一覧（Topic 一覧、画像つき）
- [ ] 弾スレッド詳細（募集一覧 ＋ 投稿フォーム）
- [ ] 投稿フォーム（欲しいカード選択・出せるカード手入力・備考・ニックネーム）
- [ ] トレード成立ボタン → 非表示、成立済みトグルで閲覧

ここで初めてブラウザで触れるものになる。触ってみて出てくる改善を反映する。

### Phase 3 — ログイン機能

### Phase 4 — 検索・絞り込み

### Phase 5 — 拡張（あったら嬉しい）

- [ ] 弾の画像を追加（slug 経由で app/assets/images/sets/ に置く）

### Phase 6 — インターネット公開

- [ ] Render 等にデプロイ

- [ ] robots.txt を選択的に緩める（一覧は載せてよいが ID ページは塞ぐ）
- [ ] 公開前に master.key をパスワードマネージャ等に控える

## 決定事項（会話で確定したこと）

| # | 決めたこと | 理由 |
|---|---|---|
| 1 | Rails 8.1.3 / Ruby 3.4.10 / SQLite / Docker | 学習しやすさ。ホストを汚さない |
| 2 | ポートはホスト側 3001 | 3000 は別プロジェクトが使用中 |
| 3 | 欲しい／譲れるを1投稿に統合 | トレードは求と出がセット。カテゴリ分けだと結びつかない |
| 4 | 弾＝スレッド、運営が事前に立てる | 過疎に見えるのを防ぐ。育ったら Board に昇格できる |
| 5 | 欲しいカードは選択式、出せるカードは手入力 | 欲しいは弾内(数十枚)、出せるは全弾(数千枚)から来る |
| 6 | 選手・弾・カードを正規化 | 所属クラブ・レア度の重複とズレを防ぐ |
| 7 | 同一選手・同一弾の複数カードは name で区別 | 得点記録／アシスト記録 が別カードとして存在する |
| 8 | レア度は数字1〜5、画面は★、当面4〜5だけ扱う | 大小比較・絞り込みができる。器は広く中身は絞る |
| 9 | トレード成立ボタン → 非表示、成立済みはトグルで閲覧 | 終わった募集を消しつつ、レート表として記録を残す |
| 10 | アカウントID は公開しない | ログイン前は晒しに悪用される。成立済みでは特に不要 |
| 11 | ログインは投稿必須・閲覧自由。後付け | 気軽さと荒らし対策の両立。Rails 8 ジェネレータで後から足せる |
| 12 | カード画像は弾ごとに1枚、自前で用意 | 枚数が激減しコスト消滅。slug 経由で後差し込み |
| 13 | robots.txt 全面ブロック（当面） | 公開判断まで検索インデックスを避ける |
| 14 | 連投対策は `rate_limit` | 構造化フォームだけでは荒らしを防げない |

## 未決事項（企画書の宿題）

作りながら決めてよいが、Phase 6（公開）までには判断が要るもの。

- **公開範囲**: 自分の練習用か、実際にネット公開して他の人にも使ってもらうか
  → これが下記すべての前提になる
- **DB**: SQLite のままか PostgreSQL に移行するか（公開して投稿を残すなら要判断）
- **カード画像の著作権**: 弾のバナー画像は B.LEAGUE の著作物。公開するなら
  ガイドライン確認または問い合わせが要る。当面はテキスト運用で先送り可能
- **不適切な投稿への対応**: 通報・削除の仕組みをどこまで作るか
- **トレードの安全性**: 成立後の評価・信頼の仕組みを入れるか

- `config/master.key` は git 管理外。**失うと credentials.yml.enc を復号できない**。
  今は中身が secret_key_base のみなので実害は小さいが、本物の秘密を入れる前・
  公開する前に安全な場所へ控えること。
- ホーム `~/` は以前 sample_app の git リポジトリになっていたが 2026-07-16 に解消済み。
  sample_app は `~/projects/sample_app` に移設。このプロジェクトとは無関係。
