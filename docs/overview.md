# TRADE COURT — 企画とこれまでの流れ

## ひとことで言うと

Bリーグカード（B.LEAGUE 公式のデジタルカードコレクション）をやっている人どうしが、
カードのトレード相手を見つけるための掲示板。
ダブったカードを無駄にせず、欲しい人へ橋渡しする「カードの物々交換所」。

## なぜ作るのか

Bリーグカードは基本ガチャで集めるが、狙ったカードが出ないことが多い。
そこで用意されているのがユーザー同士のトレード。
でも「誰が何を欲しがっていて、誰が何を譲れるのか」を知る場所がないと、相手が見つからない。
そこを掲示板でつなぐのが、このアプリの出発点。

## 想定ユーザー

- Bリーグカードをプレイしていて、欲しいカードがガチャで出なくて困っている人
- ダブったカード・手放してもいいカードを活用したい人
- 気軽にトレード相手を探したい、Bリーグ／バスケ好きなファン

## 解決したい困りごと

- 欲しいカードを持っている相手が見つからない
- 自分のダブりカードを欲しがっている人に出会えない
- トレードの募集や交渉を書き込める、共通の場所がない

## これまでの流れ（2026-07-16〜19）

設計を会話で詰めてから実装に入る形で進めてきた。
主要な論点と判断は [roadmap.md](roadmap.md) の「決定事項」にまとめてある。
ここでは大きな流れだけ記す。

### 1. 環境構築

Rails 8.1.3 / Ruby 3.4.10 / SQLite / Docker。ホスト側に Ruby / Rails を入れず、
すべて Docker の中で動かす。`docker compose up` → http://localhost:3001

### 2. 設計の作り込み

当初の素朴な企画（カテゴリ → スレッド → レス）から、次のように洗練された。

- **欲しい／譲れるを1つの投稿に統合**（カテゴリで分けると結びつかないため）
- **弾（カードセット）を掲示板の単位**にし、運営が事前に用意（過疎対策）
- **選手・弾・カードを正規化**し、情報の重複を排除
- **チームも掲示板の入口**に（弾と対等のモデルにした）
- **トレード成立ボタン**で募集を閉じ、成立記録はレート表として残す
- **アカウントIDは公開しない**（ログイン必須・本人のみ表示）

### 3. モデルとマスタデータ

Board / Topic / Team / Player / Card / Post / Wish / Offer を実装。
`db/seeds/card_sets.yml` から検証つき・冪等にデータ投入する仕組みを用意した。

作りながら、次の2点を後から見直している。どちらもデータが入る前だったので安く直せた。

- 欲しいカードを1枚 → **複数**に（出せるカードと対称にした）
- カードの管理を選手ごと → **弾ごとのラインナップ**に（データが弾単位で届くため）

### 4. 画面

弾／チームの一覧と掲示板、投稿フォーム、成立ボタンまで実装。
触ってみて出てきた改善（カード表示の整理、モーダル化、固定ヘッダー、トースト通知）も反映した。
サイト名を **TRADE COURT** とし、ロゴを配置。

### 5. 実データ（進行中）

弾 77・チーム 40・選手 371 を投入済み。カードは OCEAN の 26 枚まで。
残りの弾のカードと画像を、少しずつ追加していく。
